# Logging spec

> Status: draft, May 2026. Target phase: Phase 0/1 (applies from the first container). Cross-cutting; touches every OSCAR component.

## Two tracks

| Track | Where | What for | Read with |
|---|---|---|---|
| **Operational** | Container stdout as structured JSON ‚Üí Podman/Quadlet ‚Üí journald | Errors, lifecycle, latencies, stack traces ‚Äî "what fell over, when, in what order" | ServiceBay-MCP `get_container_logs(id, since)` / `get_service_logs(name)` / `get_podman_logs` |
| **Domain audit** | Postgres tables in `oscar-brain` (`cloud_audit`, later `gatekeeper_decisions`, `ingestion_classifications`, ‚Ä¶) | Structured accountability; retention policy; cross-call aggregation | HERMES skill `audit.query(stream, since, uid, event, ‚Ä¶)` |

Connecting element: `trace_id` (UUID per conversation turn, generated by the gatekeeper when voice audio arrives or by the gateway when a Signal/Telegram message arrives). Propagated through every MCP call and recorded in every log/audit row. In debug mode it's the precondition for assigning the STT latency to a cloud call that happened two skill steps later.

## Shared library instead of schema doc (L1)

A Python module `shared/oscar_logging/` in the repo, imported by every OSCAR container. Enforces the schema at write time instead of preaching it via README:

```python
from oscar_logging import log

log.info("router.decision",
         trace_id=ctx.trace_id, uid=ctx.uid,
         decision="cloud", router_score=0.83, reason="multi-step-plan")
```

Standard fields are filled by the library call: `ts`, `level`, `component` (from env var `OSCAR_COMPONENT`), `event` (first positional), body as keywords. The output is one JSON line per call on stdout. No more direct `print()` / `logging.info(...)` in component code.

Reason: five containers all having to call the field "trace_id" the same way is README torture. A `log.event(...)` library is more robust against drift.

## Log levels (L5)

| Level | When | Default mode | Debug mode |
|---|---|---|---|
| `error` | Failure ‚Äî operation could not complete | always | always |
| `warn` | Unusual but not a failure (cloud-connector fallback, STT confidence swing) | always | always |
| `info` | Normal lifecycle, skill calls, MCP calls (no bodies) | default | always |
| `debug` | Bodies, intermediate scores, tool args, connector request/response contents | off | on |

`debug_mode.active=true` ([oscar-architecture.md ‚Äî cross-cutting: debug mode](../oscar-architecture.md)) opens the `debug` level. Components re-query `debug_mode` on every log call (no caching > 5 s).

## Retention of domain-audit tables (L2)

| Table | Phase | Retention (`debug_mode=false`) | `debug_mode=true` |
|---|---|---|---|
| `cloud_audit` | 1 | **90 days** | unbounded |
| `gatekeeper_decisions` | 2+ | **30 days** | unbounded |
| `ingestion_classifications` | 3a+ | **180 days** (rarely written, long forensic value on misclassification) | unbounded |

Pruning: a nightly Postgres job (cron in the `oscar-brain` pod). Skipped in debug mode.

## Reading interface (L3)

**Operational logs** are not read through OSCAR ‚Äî they go straight through ServiceBay-MCP. HERMES doesn't need a wrapper skill; when the user asks "why was OSCAR slow earlier?", HERMES calls `get_container_logs` with the right filter.

**Domain audit** through *one* generic HERMES skill:

```
audit.query(
  stream: 'cloud_audit' | 'gatekeeper_decisions' | 'ingestion_classifications',
  since: ISO-8601 datetime | null,
  uid: str | null,
  event: str | null,
  limit: int = 50
)
```

Convenience phrases like "What did the cloud connector send today?" get mapped to filters by the LLM (`stream='cloud_audit', since=today-00:00`). No mini-skill per stream ‚Äî that would mean a proliferation of skills all doing the same pattern.

## PII in metadata mode (L4)

`cloud_audit` (and analogous tables) at `debug_mode=false` store **no full text**, only:

- `prompt_hash` ‚Äî SHA-256 over the full prompt. Acts as a "same as earlier" indicator across the history; no reverse possible.
- `prompt_length`, `response_length` ‚Äî tokens, not characters
- `vendor`, `cost`, `latency_ms`, `router_score`, `escalation_reason`, `uid`, `trace_id`, `ts`

**No snippet** (first 50 chars of the prompt). Half PII, half useless: not enough to debug a query, but enough to accidentally retain private content. Anyone who wants to debug full text turns on `debug_mode` ‚Äî that's the clean path.

## ServiceBay integration (L6)

Operational logs are pulled through ServiceBay-MCP ‚Äî three `read`-scope tools are available:

- **`get_container_logs(id, since)`** ‚Äî primary path. Tail pattern with `since` (Unix seconds) for debug loops; each follow-up iteration returns only new lines.
- **`get_service_logs(name)`** ‚Äî at the service level (= ServiceBay stack service name, not container id)
- **`get_podman_logs`** ‚Äî generic Podman tail, for containers without a ServiceBay service mapping

HERMES gets `read+lifecycle` bearer initially ([oscar-architecture.md](../oscar-architecture.md)) ‚Äî all three read tools accessible, no scope upgrade required.

**Secret redaction by ServiceBay:** [`src/lib/mcp/redact.ts`](https://github.com/mdopp/servicebay/blob/main/src/lib/mcp/redact.ts) removes known secret patterns (passwords, API keys, OIDC client secrets) before logs reach the MCP client. OSCAR-side stdout scrubbing is unnecessary as long as ServiceBay-MCP is the only read path. Direct host `journalctl` bypasses this layer ‚Äî emergencies only.

**ServiceBay-MCP audit as a third track:** ServiceBay records every MCP call server-side in `mcp-audit.log`. Helps with "why did HERMES restart that container?" without OSCAR auditing itself.

## Deliberately not now

- **Loki / Vector / Promtail.** No log aggregator of our own in Phase 0/1. Only once cross-container time correlation through ServiceBay-MCP gets too tedious (probably Phase 3+ with ingestion trails).
- **Web UI for logs in OSCAR.** ServiceBay has a log-viewer component ([`src/components/LogViewer.tsx`](https://github.com/mdopp/servicebay/blob/main/src/components/LogViewer.tsx)); that's enough.
- **Per-component `verbose` flags.** There's only the global `debug_mode` (cross-cutting: debug mode).
